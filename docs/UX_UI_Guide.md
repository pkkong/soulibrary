# Soulib UX/UI 디자인 가이드

Soulib은 모바일 사용자가 빠르게 책을 검색하고, 어느 전자도서관에서 이용할 수 있는지 확인하는 도구다. 화면은 iPhone 기본 앱처럼 조용하고 단순해야 한다.

## 디자인 원칙
- Mobile first: 360px 폭에서도 검색, 결과 확인, 상세 이동이 한 손 조작으로 가능해야 한다.
- One app shell: 랜딩, 검색, 상세, 신고 등 사용자-facing 페이지는 모두 같은 최대 폭 토큰을 사용한다. 현재 기준은 `--app-shell-w: 860px`이다.
- Mobile first, not mobile only: 430px 안팎의 휴대폰 화면에서는 한 손 조작에 맞춘 단일 열 UI를 유지하되, PC에서는 같은 폭 토큰 안에서 정보 밀도를 높인다.
- Desktop consistency: 데스크톱에서도 페이지 폭이 제각각이면 안 된다. 상세 페이지는 커버와 책 정보를 좌우로 배치하고, 검색/신고/홈도 같은 컨테이너 폭 체계를 따른다.
- Apple style: 흰 배경, 높은 여백 밀도, 얕은 계층, 부드러운 반경, 최소 색상만 사용한다.
- Content first: 장식보다 검색어 입력, 결과 목록, 대출 상태를 우선한다.
- Progressive feedback: 실시간 조회는 전체 완료를 기다리지 않고 카드별 상태를 먼저 보여준다.
- Fast path first: 검색 결과에서 상세 페이지로 이동할 때는 검색 응답에 포함된 책 데이터를 우선 사용한다. 사용자가 이미 본 결과를 상세에서 다시 검색하면 안 된다.
- No demo noise: 랜딩에는 플랫폼 뱃지, 가짜 예시 결과, 기능 설명 카드 등을 두지 않는다.

## 타이포그래피
- 기본 폰트: Pretendard.
- fallback: `-apple-system`, `BlinkMacSystemFont`, `Segoe UI`, `Noto Sans KR`, `sans-serif`.
- 화면 제목은 굵게 쓰되 과장하지 않는다. 모바일 앱 안의 제목은 15-19px 중심으로 유지하고, 데스크톱 상세 헤더는 26px 안팎까지 허용한다.
- 버튼, 배지, 메타 텍스트는 11-14px 범위에서 사용한다.
- `letter-spacing`은 0 또는 아주 미세한 값만 사용한다. viewport 기반 글자 크기 스케일링은 금지한다.

## 컬러
- 기본 배경: `#ffffff`.
- iOS control fill: `#f2f2f7`.
- 주요 텍스트: `#1d1d1f`.
- 보조 텍스트: `#6e6e73`.
- 경계선: `#e5e5ea` 또는 더 연한 `#f2f2f7`.
- 강조색은 최소화한다. 공급사 자체 로고 이미지를 제외하면 화면 전체가 특정 색으로 읽히지 않아야 한다.

## 랜딩 페이지
- 목적: 사용자가 바로 검색을 시작하게 만드는 것.
- 유지 요소: 서비스 목적을 설명하는 제목, 아주 짧은 설명, 검색바.
- 금지 요소: 검색바 하단 플랫폼 뱃지, 하드코딩된 예시 검색 결과, 별도 상단 검색 버튼.
- 랜딩은 설명 페이지가 아니라 검색 진입 화면이다. 정식 로고가 확정되기 전까지 중앙 브랜드 워드마크는 두지 않는다.
- 로고가 확정되면 화면 중앙이 아니라 좌상단에 작은 브랜드 표시로 붙인다.
- Soulib 심볼은 검색바 안쪽의 진입 표시로 작게 사용할 수 있다.
- 문구는 짧지만 서비스 범위를 알려야 한다. 예: "서울 전자도서관 통합검색", "교보·YES24·서울 공공 전자도서관을 함께 검색합니다."
- 아이콘과 Soulib 텍스트를 크게 붙인 이미지 lockup은 피한다. 앱 안에서는 브랜드보다 검색 목적이 먼저 보여야 한다.

## 검색 페이지
- 하단 내비게이션의 검색 아이콘은 중앙에 유지한다. 검색 후 초기 화면으로 돌아갈 수 있도록 왼쪽에 홈 버튼을 둔다.
- 오른쪽에는 오류 신고처럼 보조적인 서비스를 둘 수 있다.
- 향후 서비스가 추가되면 검색 아이콘 좌우에 배치하되, 검색은 계속 가장 강한 중앙 액션으로 유지한다.
- 검색 결과는 데스크톱에서도 무리한 다단 카드 레이아웃으로 바꾸지 않는다. 다만 컨테이너 폭은 상세/신고/홈과 같은 기준을 따른다.
- 검색 결과 카드에는 공급사/도서관 수량 뱃지를 표시하지 않는다. 검색어가 넓을 때 카드 숫자가 불완전해질 수 있으므로, 정확한 보유 수와 공급사 분류는 상세 페이지에서만 보여준다.
- 검색 결과 카드는 iOS 리스트 셀처럼 카드 전체가 상세 진입 영역이어야 한다. 상세 이동용 chevron, `상세 보기` 텍스트 버튼, 별도 원형 버튼은 쓰지 않는다.
- 검색 결과에서 저장 같은 보조 액션이 필요하면 커버 위에 올리지 않는다. 책 표지는 정보이므로 가리면 안 된다. 보조 액션은 카드 우측의 단일 icon-only 버튼으로 둔다.
- 검색 결과 공급사 분류는 상세 페이지와 필터에서 반드시 3개만 사용한다: `교보`, `YES24`, `기타`.
- 북큐브, 서울도서관, 서울교육청, 강남/은평 등은 모두 `기타`로 묶는다.
- 필터의 공급사 분류도 결과 카드와 같은 3분류 기준을 따른다. 카드와 필터가 서로 다른 분류 체계를 보여주면 안 된다.
- 검색 탭에 처음 들어왔을 때는 빈 화면처럼 보이지 않도록 짧은 검색 유도 상태를 보여준다. 단, 추천 검색어/가짜 결과/홍보 문구는 넣지 않는다.
- 두꺼운 회색 구분선이나 섹션 바를 쓰지 않는다. 필요한 구분은 여백, 미세한 경계선, 카드 간 간격으로 처리한다.
- 결과 카드는 모바일 목록형으로 유지한다: 좌측 커버, 가운데 제목/저자/출판사, 우측 책갈피 보조 액션.

## 상세 페이지
- 책 상세 상단의 커버 중심 구성은 유지한다.
- 상세 페이지는 모바일과 PC 모두 같은 정보 흐름을 유지한다: 커버와 흐린 배경, 제목, 저자, 출판사, 소장 도서관 순서.
- PC에서는 같은 흐름을 유지하되 앱 셸 폭과 내부 여백만 넓혀 답답하지 않게 만든다. 커버와 제목 정보를 좌우로 나누지 않는다.
- 내 서재에 담기는 책갈피 아이콘으로 처리한다. 상세 페이지 상단과 검색 결과 카드 우측에서 같은 아이콘 언어를 사용한다.
- 검색 결과에서 상세로 들어온 경우에는 실시간 검색 결과 캐시를 사용해 상세 화면을 즉시 연다.
- 소장 도서관 상태 조회는 카드 자체를 숨기지 않는다. 카드와 도서관명은 즉시 보여주고, 상태 텍스트 자리에서만 로딩을 표시한다.
- 상태 정렬은 조회 완료 후 부드럽게 반영한다. 사용자가 빈 영역만 보는 시간이 생기면 안 된다.

## 내 서재
- 내 서재는 보조 서비스지만 하단 내비게이션에서 바로 접근할 수 있어야 한다.
- 검색 결과 카드의 단순한 상세 진입 흐름을 깨지 않기 위해, 책 담기는 카드 우측의 작은 책갈피 버튼으로만 제공한다. 커버 위 오버레이, 텍스트 버튼, 검은 원형 버튼은 쓰지 않는다.
- 서재 담기 UI는 하단 sheet를 사용한다. 새 서재 생성은 입력창과 `+` 아이콘 버튼으로만 처리하고, 긴 설명 문구를 넣지 않는다.
- 내 서재 화면의 새 서재 입력창은 기본으로 접어둔다. 목록을 보는 화면이 관리 폼처럼 보이면 안 된다.
- 내 서재 목록은 iOS 리스트 셀처럼 보이게 한다: 좌측 커버, 가운데 제목/저자/출판사, 우측 `빼기` 보조 액션. 상세 이동은 카드 전체 탭으로 처리하고 별도 chevron을 두지 않는다.
- 서재에서 책을 빼는 동작은 텍스트 버튼이 아니라 작은 icon-only 버튼으로 둔다. 단, `aria-label`은 반드시 제공한다.
- 내 서재의 선택 칩, 빈 상태 액션, 생성 버튼은 검은색을 기본값으로 쓰지 않는다. 검은색은 중앙 검색 탭이나 정말 강한 1차 액션에만 제한한다.
- 저장된 책의 소장 수/공급사 수는 검색어와 저장 시점에 따라 어긋날 수 있으므로 내 서재 카드에 강조하지 않는다. 정확한 소장 상태는 상세 페이지에서 다시 확인한다.

## 컴포넌트 규칙
- 검색바: `#f2f2f7` 배경, 16px 전후 radius, icon + input 구조.
- 필터 버튼: 검색바와 같은 시스템 필 색상을 사용하고, 텍스트보다 아이콘 중심으로 둔다.
- 필터 시트: 하단 sheet 형태, 24px 상단 radius, 흐린 오버레이.
- 배지/칩: 플랫폼을 설명하기 위한 장식 칩을 남발하지 않는다. 결과 요약은 카운트 중심으로만 보여준다.
- 카드 안에 카드를 넣지 않는다.

## 앱 아이콘
- iOS 홈 화면용 `apple-touch-icon.png`는 반드시 투명도 없는 정사각형 PNG로 관리한다.
- 로고는 iOS 둥근 마스크에서 잘리지 않도록 전체 캔버스의 65-70% 안쪽에 배치한다.
- Safari 캐시가 강하므로 아이콘 교체 시 `base.html`의 icon query version도 함께 올린다.
- PWA/Android용 manifest 아이콘은 `app-icon-192.png`, `app-icon-512.png`를 사용한다.

## 카피라이팅
- AI스럽거나 마케팅처럼 들리는 문장을 피한다.
- 사용자가 지금 할 행동을 설명하는 짧은 문장을 쓴다.
- "한눈에", "쉽고 빠르게", "어디서 빌릴지 확인"처럼 범용적인 홍보 문구는 필요한 경우가 아니면 피한다.

## 접근성
- 검색 input에는 명시적 label 또는 `sr-only` label을 둔다.
- icon-only 버튼은 `aria-label`을 반드시 둔다.
- 모바일 키보드 검색 액션을 위해 검색 input에는 `type="search"`와 `enterkeyhint="search"`를 사용한다.
- 로딩 상태는 전체 영역을 막지 않고, 상태가 바뀌는 요소 가까이에 표시한다.

## 오류 신고
- 오류 신고는 하단 내비게이션 오른쪽 보조 액션으로 둔다.
- 신고 폼은 분류, 내용, 문제가 있던 주소, 선택 연락처만 받는다.
- 연락처는 목록에 노출하지 않는다. 공개 목록에는 분류, 접수 시각, 내용만 표시한다.
- 제출 후에는 성공 메시지만 보여주지 말고, 방금 접수된 신고 카드를 즉시 노출한다. GitHub Issues 목록 동기화가 늦어도 사용자는 접수 사실을 확인할 수 있어야 한다.
- 제출 버튼은 중복 클릭을 막기 위해 `보내는 중...` 상태와 disabled 상태를 표시한다.
- 오류 신고의 단일 저장소는 GitHub Issues다. DB/JSONL 파일 fallback은 사용하지 않는다.

## 주요 파일
- 랜딩 템플릿: `web/templates/index.html`
- 검색 템플릿: `web/templates/search.html`
- 일반 상세 템플릿: `web/templates/book.html`
- 실시간 상세 템플릿: `web/templates/live_book.html`
- 내 서재 템플릿: `web/templates/my_shelf.html`
- 공통 UI 스타일: `web/static/css/search.css`
- 검색 결과 렌더링: `web/static/js/search.js`
- 내 서재 저장소: `web/static/js/shelf_storage.js`
- 내 서재 화면: `web/static/js/my_shelf.js`
- 내 서재 담기 sheet: `web/static/js/shelf_picker.js`
- 실시간 검색 공급사 분류: `web/live_search/service.py`, `web/live_search/normalizer.py`
- 오류 신고 라우트: `web/report_routes.py`
- 오류 신고 템플릿: `web/templates/reports.html`
